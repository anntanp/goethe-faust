# Plan: Build `lookup_dctype_to_class.csv` — Type Dispatch for the Transform

## Context

`transform_edm_to_mocho.py` needs to assign domain-specific rdf:type subclasses to
ProvidedCHO entities based on three dimensions: mediatype (mt001–mt007), sector
(sparte001–007), and the dc:type string value. This dispatch logic existed in
`old-config/type2class.json` and `old-config/audio_type2class.json` (FRBR/FaBiO
classes, 4 WEMI levels), but was never ported to the mocho-aligned pipeline.

The current `alignment_ddbedm_mocho.csv` only covers property alignment (EDM field →
RDA property via DC→RDA map). It has no type dispatch. The htype lookup
(`lookup_htype_doco_rico.csv`) covers archival hierarchy types (htype_030–048) but
not the full sector × mediatype × dc:type space.

This plan builds a new lookup file and generator script to fill that gap. The
transform script will use it in `retype_entities()` as a second dispatch after
the htype lookup.

## 2. Output CSV schema

`lookup_dctype_to_class.csv` — one row per (mediatype, sector, dc_type_de) triple:

| Column | Description |
|---|---|
| `mediatype` | IRI (e.g. `http://ddb.vocnet.org/medientyp/mt001`) or `any` |
| `sector` | IRI (e.g. `http://ddb.vocnet.org/sparte/sparte005`) or `any` |
| `dc_type_de` | German dc:type literal string as it appears in the corpus (current lookup key) |
| `dc_type_en` | English translation (from old-config `en` field) |
| `dnb_uri` | GND Sachbegriff IRI from old-config `dnb` field (future lookup key; see §2.2) |
| `rdf_type_m` | Manifestation-level class IRI (the primary rdf:type to emit alongside mocho:Manifestation) |
| `rdf_type_e` | Expression-level class IRI (emit if present) |
| `rdf_type_w` | Work-level class IRI (emit if present; for audio: not emitted for ProvidedCHO — future Work-entity generation only) |
| `rdf_type_i` | Item-level class IRI (emit if present) |
| `source_vocab` | Which vocabulary the class belongs to: `mo`, `aco`, `vra`, `rico`, `doco`, `fabio`, `mocho` |
| `notes` | Free-text notes from old-config `remarks` field |

### 2.1 Lookup priority in the transform script

1. Exact match on (mediatype, sector, dc_type_de)
2. Match on (mediatype, `any`, dc_type_de)
3. Match on (`any`, `any`, dc_type_de)
4. No match → no additional rdf:type beyond mocho:Manifestation + htype

## 3. Class mapping: old-config → mocho-aligned

The old-config uses FRBR/FaBiO with 4 positional WEMI slots `[W, E, M, I]`.
The new lookup uses named columns and mocho-aligned vocabularies.

### 3.1 Audio (mt001) — from `audio_type2class.json`

| Old classes (M slot) | New `rdf_type_m` |
|---|---|
| `mo:Record`, `aco:AudioClip` | `aco:AudioManifestation` (general); `aco:AudioClip` only for clip/excerpt dc:types |
| `mo:MusicalManifestation` | `mo:MusicalManifestation` |
| `fabio:AudioDocument` at E + `aco:AudioClip` at M | `aco:AudioManifestation` |

Key audio mappings:
- Musical (Schallplatte, CD, Langspielplatte, Folie, Schellackplate, Schallfolie+music topic): `rdf_type_w = mo:MusicalWork`, `rdf_type_m = mo:MusicalManifestation`, `rdf_type_i = aco:AudioFile`
- Non-musical audio (Audiofile, Hörfunksendung, Inszenierung, Tondokument non-music, Magnettonband/Vortrag): `rdf_type_e = fabio:AudioDocument`, `rdf_type_m = aco:AudioManifestation`, `rdf_type_i = aco:AudioFile`
- Default audio: `rdf_type_m = aco:AudioManifestation`, `rdf_type_i = aco:AudioFile`
- Note: `aco:AudioClip` (subclass of `aco:AudioManifestation`) is reserved for dc:type values that denote clips/excerpts specifically.

### 3.2 Text (mt003) — from `type2class.json`

FaBiO is deprecated by RDA. The old-config entries are reference examples only;
do not carry FaBiO class IRIs into the new lookup. Text types are typed at the
Manifestation level via RDA classes.

Sector-dependent:

| Sector | `rdf_type_m` |
|---|---|
| sparte001 (Archive) | `rico:Record` (archival text document; htype takes precedence if present) |
| sparte002 (Library) | `rdam:C10007` (RDA Manifestation) — equivalent to `mocho:Manifestation`; no additional subclass needed at this pass |
| sparte005 (Museum) | `mocho:Manifestation` only (no RDA text subclass without VRA equivalent) |
| `any` | `rdam:C10007` |

#### 3.2.1 Specific bibliographic categories (Thesis, Textbook, etc.)

In RDA, document categories are expressed via properties, not rdf:type subclasses.
For example, a Dissertation is expressed as `rdaw:P10004 has category of work` with
a controlled value — not as `rda:Dissertation` class. Therefore:

- The dc:type string (e.g. `Dissertation`, `Lehrbuch`) is preserved as a literal
  via property alignment (dc:type → RDA property), **not** promoted to rdf:type.
- No `rdf_type_m` entry is emitted for specific bibliographic genres in the
  Text+Library sector from this lookup table.
- DoCO structural classes (`doco:Section`, `doco:Chapter`, etc.) continue to be
  emitted via the **htype dispatch** (lookup_htype_doco_rico.csv), not here.

**Future work**: Create a controlled vocabulary for DDB dc:type string values and
align it to an existing KB (e.g. GND Sachbegriff, RDA value vocabularies, or
Wikidata). This would allow dc:type literals to be resolved to LOD concept URIs,
enabling richer faceting and cross-dataset type alignment beyond the current
literal-string lookup.

### 3.3 Photo (mt002) — from `image_type2class.json` (revised 2026-04-17)

**Revised model** — see `notes/image-type-class-mapping.md` for full ADR discussion.
Previous model (vra:Image + lio:Image) replaced by group-based dc:type × sector dispatch.

| Group | dc:types | Sectors | ProvidedCHO class | WEMI slot |
|---|---|---|---|---|
| A — 2D Artworks | ARTWORK_2D (Zeichnung, Gemälde, Druckgraphik, …) | sparte005, sparte006 | `vra:Work` | W |
| B — 3D Objects | OBJECTS_3D (Skulptur, Büste, Medaille, …) | any | `vra:Work` | W |
| C — Photo Works | PHOTO_TYPES (Fotografie, Standfoto, Postkarte, …) | sparte005, sparte006 | `mocho:ImageWork` | W |
| C+ | Bild, Druck | sparte005 only | `mocho:ImageWork` | W |
| D — Architecture | ARCHITECTURE (Baudenkmal, Denkmal, Wohnhaus, …) | any | `mocho:ImmovableWork` | W |
| E — Archive photo | PHOTO_TYPES in sparte001 | sparte001 | *(skip — rico:Record via htype)* | — |
| Defaults | unmatched | sparte006, sparte003 | `vra:Work` | W |
| Defaults | unmatched | sparte005 | `mocho:ImageWork` | W |
| F — Default | all remaining | any | `mocho:Manifestation` | M |

**W-slot semantics**: For mt002 ProvidedCHOs with a W-slot class, the transform emits the
W-slot class **instead of** `mocho:Manifestation` (not in addition to it). M-slot entries
follow the existing accumulation rule (stacked on top of `mocho:Manifestation`).

**WebResources** (edm:isShownBy / edm:hasView targets): always typed as
`mocho:ImageObject` (subClassOf `mocho:Manifestation`) + `rdac:C10007` + `rdam:P30001 rdact:1018`
+ `vra:imageOf <cho-uri>` (interim; Phase B: replace with `mocho:facsimile`).

**CLASS_MAP additions for `gen_dctype_class_mapping.py`**:
```python
"vra:Work":              "http://purl.org/vra/Work",
"mocho:ImageWork":       "https://ise-fizkarlsruhe.github.io/ddbkg/mocho#ImageWork",
"mocho:ImmovableWork":   "https://ise-fizkarlsruhe.github.io/ddbkg/mocho#ImmovableWork",
"mocho:ImageObject":     "https://ise-fizkarlsruhe.github.io/ddbkg/mocho#ImageObject",
```

### 3.4 Video (mt005) — from `video_type2class.json` (new)

dc:type dispatch → EBUCore Plus classes (confirmed in mocho-full.owl; prefix `ebucoreplus`,
IRI base `http://www.ebu.ch/metadata/ontologies/ebucoreplus#`).

- Manifestation-level target: `ebucoreplus:MediaResource` (default) or `ebucoreplus:EditorialWork`
  for editorial/programme content, per dc:type value
- Sector refinement:
  - sparte004 (Mediathek): `ebucoreplus:EditorialWork` for programme types; `ebucoreplus:MediaResource` otherwise
  - `any`: `ebucoreplus:MediaResource` as default
- dc:type values and sub-mappings: populated after Q4 frequency count (task A1).

### 3.5 Not Digitized (mt007)

Follows the standard dispatch route: htype → sector → dc:type. The absence of
a digital surrogate does not exempt an object from type classification.
`mocho:Manifestation` is always emitted as the base type; additional classes
follow from htype, sector, and dc:type values as for all other mediatypes.

## 6. Dispatch priority and combination rules

Four dimensions can contribute rdf:type triples for a ProvidedCHO. Their roles
are complementary, not competing:

| Dimension | Source | Output | When |
|---|---|---|---|
| `htype` | `lookup_htype_doco_rico.csv` | DoCO class (library) or RiC-O class (archival) | htype_001–029 → DoCO; htype_030–048 → RiC-O |
| `mediatype` × `sector` × `dc:type` | `lookup_dctype_to_class.csv` | Domain class (ACO/MO/VRA/LIO/EBUCore/RiC-O) | Any ProvidedCHO with a dc:type value |
| base | hardcoded | `mocho:Manifestation` | Always |

**Combination rule**: Accumulate — both htype and dc:type dispatch can fire
for the same entity. Identical rdf:type IRIs are deduped. Both layers are
independent: an archival letter can be both `rico:Record` (from htype_034) AND
a more specific genre class (from dc:type `Brief`). Neither blocks the other.

### 6.1 Separation by htype domain

- htype_030–048 (archival): htype dispatch produces RiC-O classes; dc:type dispatch
  may also produce a more specific document genre class. Both emitted.
- htype_001–029 (library/DoCO): htype dispatch produces DoCO structural classes
  (Chapter, Section, etc.); dc:type dispatch may produce a publication type class.
  Both emitted if present.
- No htype: only dc:type dispatch contributes a specific class (beyond mocho:Manifestation).

### 6.2 mediatype gates the dispatch table

- `mt001` (Audio): rows from `audio_type2class.json` → ACO/MO classes
- `mt002` (Photo): rows from `image_type2class.json` → VRA/LIO/RiC-O classes
- `mt003` (Text): rows from `type2class.json` → RDA/RiC-O classes
- `mt005` (Video): rows from `video_type2class.json` → EBUCore Plus classes
- `mt007` (Not Digitized): follows standard dispatch — htype + sector + dc:type

### 6.3 Priority order for the transform loop (implemented in `retype_entities()`)

1. Always emit `mocho:Manifestation`
2. Apply htype dispatch (existing `lookup_htype_doco_rico.csv`)
3. Apply dc:type dispatch (new `lookup_dctype_to_class.csv`), using mediatype to gate rows
4. Dedup: collect all rdf:type IRIs in a set, emit each once

**Validation before finalising**: Run `sample_type_dispatch.py` on 50 random records
per sector × mediatype cell. Manual review determines if the accumulate model is
correct or if priority overrides are needed for specific cases.

## 7. Sampling script: `sample_type_dispatch.py`

A lightweight diagnostic script (not part of the transform pipeline) that:

1. Reads `items-all-goethe-faust.json` (streaming, random sample)
2. For each sampled record, shows:
   - object_id, sector IRI, mediatype IRI
   - dc:type value(s)
   - htype code
   - rdf:type classes emitted by htype dispatch (from lookup_htype_doco_rico.csv)
   - rdf:type classes emitted by dc:type dispatch (from lookup_dctype_to_class.csv)
   - combined rdf:type set (deduped)
3. Samples at least 5 records per populated (sector × mediatype) cell
4. Output: `output/type_dispatch_sample.csv` + printed summary

This is a prerequisite for finalising `retype_entities()` in the transform script.
